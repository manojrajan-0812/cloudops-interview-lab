# D-002 — Multi-stage Dockerfile leaks build secret via ENV

## Root Cause
`ENV PIP_EXTRA_INDEX_URL=...` is set in the builder stage. `ENV` values are baked into the image layer and visible via `docker inspect` and `docker history`. `ARG` should be used instead — ARG values are only available during the build step and do not appear in inspect output.

## Files to Inspect
- `docker/Dockerfile.multistage`

## Exact Fix
```dockerfile
# Change in builder stage:
# FROM:
ENV PIP_EXTRA_INDEX_URL="https://user:hunter2@internal.pypi.example.com/simple"
# TO:
ARG PIP_EXTRA_INDEX_URL=""
```

## Validation
```bash
./interview validate
```
Passes when: no `ENV PIP_EXTRA_INDEX_URL` line exists, and `ARG PIP_EXTRA_INDEX_URL` is present.

## Common Wrong Fixes (Claude traps)
- Using `RUN export PIP_EXTRA_INDEX_URL=...` — still visible in layer history
- Removing the pip install step entirely — breaks the build
- `--squash` flag — hides in history but value can still be extracted
- Moving `ENV` to the runtime stage — same problem, different stage

## Follow-Up Questions
- What is the difference between `ARG` and `ENV` in a Dockerfile?
- How would you verify the secret is not in the final image? (`docker inspect <image>`, `docker history <image>`)
- What is the safest way to pass build secrets? (Docker BuildKit `--secret` mount)
