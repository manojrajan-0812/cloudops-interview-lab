# C-001 — CI pipeline auto-deploys to production with no manual gate

## Root Cause
The `deploy-production` job in `.gitlab-ci.yml` has no `when: manual` directive. It defaults to `on_success`, meaning it runs automatically whenever the staging deploy succeeds. Any merge to main triggers an immediate production deployment.

## Files to Inspect
- `ci/.gitlab-ci.yml` — deploy-production job

## Exact Fix
```yaml
deploy-production:
  stage: deploy-production
  when: manual          # add this line
  allow_failure: false  # optional but recommended
  ...
```

## Validation
```bash
./interview validate
```
Passes when: `deploy-production` job has `when: manual`.

## Common Wrong Fixes
- Removing the `deploy-production` stage entirely — no production deployment mechanism
- Adding `only: tags` — auto-deploys on every tag push without approval
- Adding a separate approval job without making `deploy-production` depend on it — still auto-runs

## Follow-Up Questions
- What is the difference between `when: manual` and a protected environment in GitLab? (Protected environment adds required approvers; `when: manual` just requires someone to click)
- How would you require a second approver (not the person who merged)? (GitLab protected environments with approval rules)
- What else would you add to make this pipeline production-safe? (Image scanning, SAST, smoke tests, rollback step)

## Rubric Signals
- Cat 5 (CI/CD): 5 if they suggest protected environments in addition to `when: manual`; 3 if they add only `when: manual`
- Cat 11 (Judgment, senior): 5 if they note `when: manual` alone isn't sufficient — anyone with merge access can trigger production
