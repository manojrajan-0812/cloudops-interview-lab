# M-001 — Prometheus alert fires on every single error

## Root Cause
The `HighErrorRate` alert has three problems:
1. `expr: rate(http_requests_total{status="500"}[5m]) > 0` — fires on a single HTTP 500 error
2. No `for:` clause — pages immediately on the first occurrence with no grace period
3. No ratio calculation — counts raw errors rather than error rate as a percentage of traffic

## Files to Inspect
- `monitoring/prometheus-rules.yaml` — HighErrorRate rule

## Exact Fix
```yaml
- alert: HighErrorRate
  expr: |
    sum(rate(http_requests_total{job="interview-api", status=~"5.."}[5m]))
    /
    sum(rate(http_requests_total{job="interview-api"}[5m])) > 0.05
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "High HTTP error rate on interview-api"
    description: "Error rate is {{ $value | humanizePercentage }} over the last 5 minutes."
```

## Validation
```bash
./interview validate
```
Passes when: threshold > 0; `for:` clause is present; expression contains a division operator outside label selectors.

## Common Wrong Fixes
- Just adding `for: 5m` without changing the threshold — still fires on one error after 5 minutes
- Setting threshold to `0.5` (50%) — far too permissive; most production SLOs are 1-5%
- Removing the alert entirely — validator rejects this

## Follow-Up Questions
- What is the difference between `rate` and `irate` in PromQL?
- Why divide by total requests rather than using a raw count threshold? (Traffic volume varies; 10 errors at 10 req/s is catastrophic; at 10,000 req/s it's noise)
- How would you test a Prometheus alerting rule before deploying it? (`promtool test rules`)
