# K-009 — API key stored in ConfigMap instead of Secret

## Root Cause
`k8s/configmap.yaml` contains `API_KEY: "supersecret123"` in plaintext. ConfigMaps are not encrypted at rest by default and are readable by any pod in the namespace with ConfigMap read access. Secrets are the correct resource for sensitive values.

## Files to Inspect
- `k8s/configmap.yaml` — contains the API_KEY
- `k8s/secret.yaml` — currently missing the API_KEY
- `k8s/deployment.yaml` — envFrom references (already correct in baseline)

## Exact Fix
1. Remove `API_KEY` from configmap.yaml
2. Add to secret.yaml:
```yaml
data:
  API_KEY: c3VwZXJzZWNyZXQxMjM=   # base64 of "supersecret123"
```
Or use `stringData` (Kubernetes encodes it automatically):
```yaml
stringData:
  API_KEY: "supersecret123"
```
The deployment's `secretRef` already points to `interview-api-secret` so no deployment change is needed.

## Validation
```bash
./interview validate
```
Passes when: `API_KEY` is absent from ConfigMap; `API_KEY` is present in Secret (either `data` or `stringData`).

## Common Wrong Fixes
- Leaving API_KEY in ConfigMap AND adding it to Secret — duplication; validator catches ConfigMap still has it
- Hardcoding the key directly in the deployment `env` block — worse than ConfigMap
- Base64-encoding the value but leaving it in the ConfigMap — still a ConfigMap, not encrypted

## Follow-Up Questions
- Are Kubernetes Secrets actually secret? What are their limitations? (Base64 only; need encryption at rest + RBAC)
- How do tools like Vault or Sealed Secrets improve on native Secrets?
- How would you rotate this key without redeploying the pod?
