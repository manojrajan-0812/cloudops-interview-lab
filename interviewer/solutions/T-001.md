# T-001 — Terraform timestamp() trigger causes destroy+recreate every plan

## Root Cause
`null_resource.deployment` has `triggers = { always_run = timestamp() }`. `timestamp()` is re-evaluated on every `terraform plan`, so the trigger value always changes, causing Terraform to plan a destroy+recreate of the resource on every run.

## Files to Inspect
- `terraform/main.tf` — null_resource.deployment triggers block
- Run `terraform plan` to observe the destroy+recreate in the output

## Exact Fix
Replace `timestamp()` with a stable value:
```hcl
resource "null_resource" "deployment" {
  triggers = {
    app_version  = var.app_version
    environment  = var.environment
  }
  ...
}
```

## Validation
```bash
./interview validate
```
Passes when: no `timestamp()` call exists in `main.tf`; the `null_resource` still exists.

## Common Wrong Fixes
- `lifecycle { ignore_changes = [triggers] }` — hides the problem rather than fixing it
- Removing the `null_resource` entirely — wrong fix; resource may be needed for provisioners
- Replacing `timestamp()` with `uuid()` — also volatile, same problem

## Follow-Up Questions
- What is the purpose of `triggers` in a `null_resource`?
- When is destroy+recreate acceptable in Terraform? (Ephemeral resources, test environments)
- How would you catch this in CI before it hits production? (`terraform plan -detailed-exitcode`, check output for `must be replaced`)
