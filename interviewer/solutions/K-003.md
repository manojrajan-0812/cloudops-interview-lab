# K-003 — Deployment missing resource limits

## Root Cause
The deployment has no `resources:` block. Without limits, the pod can consume all available node memory and CPU, causing OOMKill of itself or other pods on the node.

## Files to Inspect
- `k8s/deployment.yaml` — containers section (no resources block)

## Exact Fix
```yaml
resources:
  requests:
    cpu: "100m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "256Mi"
```
Both `requests` AND `limits` are required. The values are flexible — the validator checks that both `cpu` and `memory` are set for each.

## Validation
```bash
./interview validate
```
Passes when all containers have `resources.requests.cpu`, `resources.requests.memory`, `resources.limits.cpu`, and `resources.limits.memory`.

## Common Wrong Fixes
- Adding only `limits` without `requests` — scheduler can't plan placement correctly
- Setting limits to `0` or leaving values empty — validator rejects
- Very low limits (e.g. `cpu: "1m"`) — technically passes but would cause throttling

## Follow-Up Questions
- What is the difference between `requests` and `limits`?
- What happens when a pod exceeds its memory limit? (OOMKilled by kernel)
- How do you size requests and limits for a service you've never profiled? (Start with defaults, use VPA or metrics history)

## Rubric Signals
- Cat 3 (Kubernetes): 5 if they add both requests AND limits immediately; 3 if they need a hint that both are required
- Cat 10 (Communication): 5 if they explain the OOM kill mechanism, not just "it uses too much memory"
