# K-008 — Pod runs privileged as root

## Root Cause
The deployment sets `runAsUser: 0` (root) at pod level and `privileged: true` with `allowPrivilegeEscalation: true` at container level. None of these are needed by the application.

## Files to Inspect
- `k8s/deployment.yaml` — pod securityContext and container securityContext

## Exact Fix
```yaml
# Pod-level securityContext
securityContext:
  runAsNonRoot: true
  runAsUser: 1000
  fsGroup: 1000

# Container-level securityContext
securityContext:
  privileged: false          # remove or set to false
  allowPrivilegeEscalation: false
  capabilities:
    drop: ["ALL"]
```

## Validation
```bash
./interview validate
```
Passes when: `runAsUser != 0`; `allowPrivilegeEscalation: false`; `capabilities.drop` contains `ALL`.

## Common Wrong Fixes (Claude traps)
- **Only setting `runAsNonRoot: true` while leaving `privileged: true`** — still dangerous; validator catches it
- **Setting `runAsUser: 65534` (nobody) without removing `privileged`** — partial fix
- **Removing the securityContext block entirely** — defaults are insecure; validator fails

## Follow-Up Questions
- What can an attacker do with a privileged container that they can't do otherwise? (Escape to host, mount host filesystem, access other containers)
- What does `allowPrivilegeEscalation: false` prevent? (Prevents processes inside the container from gaining more privileges than their parent)
- How would you enforce this across all deployments? (PodSecurityAdmission, OPA Gatekeeper)

## Rubric Signals
- Cat 3 (Kubernetes): 5 if they know to fix all three fields; 3 if they fix only root user
- Cat 11 (Judgment, senior): Claude often suggests only `runAsNonRoot: true` — strong candidate fixes all three layers
