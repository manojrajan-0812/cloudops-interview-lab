# K-006 — No PodDisruptionBudget

## Root Cause
The deployment has no `PodDisruptionBudget`. During a node drain or cluster upgrade, the eviction controller can remove all replicas simultaneously, causing a full service outage. A PDB prevents this by constraining how many pods can be voluntarily disrupted at once.

## Files to Inspect
- `k8s/pdb.yaml` — currently empty/placeholder
- `k8s/deployment.yaml` — replicas: 2 (confirms why minAvailable: 1 matters)

## Exact Fix
```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: interview-api-pdb
  namespace: default
spec:
  minAvailable: 1
  selector:
    matchLabels:
      app: interview-api
```

## Validation
```bash
./interview validate
```
Passes when: a PodDisruptionBudget exists with `minAvailable >= 1` and `selector.matchLabels` targeting the deployment.

## Common Wrong Fixes (Claude traps)
- **Increasing `maxSurge` in the Deployment** — controls rolling deployment speed, not node eviction
- **Setting `maxUnavailable: 0` in rollingUpdate** — controls deployments only, not `kubectl drain`
- **Increasing replicas to 3 without a PDB** — still drainable all at once
- **PDB with wrong selector** — validator catches this; PDB doesn't protect the right pods

## Follow-Up Questions
- What is the difference between a PDB and `rollingUpdate` settings?
- When does a PDB NOT protect you? (Hardware failure — PDB only governs *voluntary* disruptions)
- What value of `minAvailable` would you choose for a 2-replica deployment? (1 — allows rolling maintenance while keeping service available)
