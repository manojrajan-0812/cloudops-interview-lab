# K-005 — /health always returns 200 even when DB is down

## Root Cause
The `/health` handler in `app/app.py` returns `200` unconditionally without checking whether the database is reachable. Kubernetes keeps routing traffic to the pod even when the DB is down because the readiness probe passes.

## Files to Inspect
- `app/app.py` — the `/health` route handler

## Exact Fix
```python
@app.route("/health")
def health():
    try:
        conn = get_db()
        conn.execute("SELECT 1")
        conn.close()
        return jsonify({"status": "ok", "db": "connected"}), 200
    except Exception as e:
        logger.error(f"Health check failed: {e}")
        return jsonify({"status": "degraded", "db": "unreachable", "error": str(e)}), 503
```

## Validation
```bash
./interview validate
```
Validator runs two tests: `/health` with a reachable DB must return 200; with an unreachable DB path must return 503.

## Common Wrong Fixes (Claude traps — this is the senior test)
- **Adding `/healthz` that always returns 200 and pointing the probe there** — masks the real bug; readiness probe becomes meaningless
- **Removing the readiness probe** — pods stay in rotation permanently regardless of DB state
- **Catching the exception but still returning 200** — looks like a fix, still broken
- **Only checking if the DB file exists** — doesn't verify the connection works

## Follow-Up Questions
- Why is a cargo-cult health check dangerous in Kubernetes specifically? (Readiness probe is how k8s removes unhealthy pods from the load balancer — if it lies, traffic goes to broken pods)
- Should the liveness probe also check the DB? (Generally no — liveness should only restart truly deadlocked processes; readiness handles dependency issues)
- What would you monitor to detect this pattern in a code review? (Any /health handler that doesn't have a dependency check)
