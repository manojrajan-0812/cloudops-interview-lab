# K-001 — Liveness probe causes restart loop on slow-starting pod

## Root Cause
`livenessProbe.initialDelaySeconds: 1` fires before the app finishes its ~15-second startup. `failureThreshold: 1` means a single missed check immediately kills the pod. The pod restarts in a loop before it can serve traffic.

## Files to Inspect
- `k8s/deployment.yaml` — livenessProbe section
- `kubectl describe pod <name>` — Events section shows "Liveness probe failed"

## Exact Fix
```yaml
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 30   # was 1
  periodSeconds: 15
  failureThreshold: 3        # was 1
```

## Validation
```bash
./interview validate
```
Passes when: `initialDelaySeconds >= 20` and `failureThreshold >= 3`.

## Common Wrong Fixes
- Removing the `livenessProbe` entirely — pod is never restarted on real deadlock; validator rejects this
- Fixing only `initialDelaySeconds` but not `failureThreshold` — validator still fails
- Modifying `readinessProbe` instead of `livenessProbe` — different field, different effect

## Follow-Up Questions
- What is the difference between liveness and readiness probes?
- What happens if you remove the liveness probe entirely?
- How would you tune these values for a service you don't own? (Measure actual startup time from logs)
