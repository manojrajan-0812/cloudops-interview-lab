# K-007 — ServiceAccount bound to cluster-admin

## Root Cause
`k8s/rbac.yaml` has a `ClusterRoleBinding` that binds `interview-api-sa` to the `cluster-admin` ClusterRole. The application only needs to read ConfigMaps in one namespace — `cluster-admin` grants full control of the entire cluster.

## Files to Inspect
- `k8s/rbac.yaml` — ClusterRoleBinding section

## Exact Fix
Remove the ClusterRoleBinding. Add a namespace-scoped Role and RoleBinding:
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: interview-api-role
  namespace: default
rules:
  - apiGroups: [""]
    resources: ["configmaps"]
    verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: interview-api-rolebinding
  namespace: default
subjects:
  - kind: ServiceAccount
    name: interview-api-sa
    namespace: default
roleRef:
  kind: Role
  apiGroup: rbac.authorization.k8s.io
  name: interview-api-role
```

## Validation
```bash
./interview validate
```
Passes when: no ClusterRoleBinding to cluster-admin exists for the SA; a Role with non-wildcard verbs exists; a RoleBinding connects `interview-api-sa` to that Role.

## Common Wrong Fixes (Claude traps)
- **Creating a Role with `verbs: ["*"]`** — still over-privileged; validator catches it
- **Creating a ClusterRole instead of Role** — namespace scope is correct here
- **Binding a different SA** — validator checks the RoleBinding subjects explicitly
- **Removing the SA entirely** — deployment references it; breaks the pod spec
