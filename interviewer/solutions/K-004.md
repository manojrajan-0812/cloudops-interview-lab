# K-004 — Ingress routes to wrong service port

## Root Cause
`k8s/ingress.yaml` specifies backend port `8080` (the container port) but the Service only exposes port `80` (which internally routes to container port 8080 via `targetPort`). The Ingress must reference the Service's port, not the container's port.

## Files to Inspect
- `k8s/ingress.yaml` — spec.rules.http.paths.backend.service.port.number
- `k8s/service.yaml` — spec.ports (port: 80, targetPort: 8080)

## Exact Fix
```yaml
# k8s/ingress.yaml
backend:
  service:
    name: interview-api-svc
    port:
      number: 80   # was: 8080
```

## Validation
```bash
./interview validate
```
Passes when the Ingress backend port matches a port exposed by the Service.

## Common Wrong Fixes
- Changing `service.yaml` port to 8080 — breaks the 80→8080 mapping convention
- Changing `targetPort` to 80 — app listens on 8080, this would break routing
- Adding a new Service on port 8080 — unnecessary complexity

## Follow-Up Questions
- What is the difference between `port`, `targetPort`, and `nodePort` in a Service?
- How does an Ingress controller resolve backend ports? (It sends traffic to the Service's `port`, which routes via `targetPort` to the pod)
- How would you debug this in production? (`kubectl describe ingress`, check ingress controller logs)
