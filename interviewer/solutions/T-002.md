# T-002 — Terraform backend is local (no remote state or locking)

## Root Cause
`terraform/backend.tf` uses `backend "local"`. State is stored on the engineer's local machine. Two engineers running `terraform apply` concurrently will corrupt state. No team member can see current state without the file.

## Files to Inspect
- `terraform/backend.tf`

## Exact Fix
```hcl
terraform {
  backend "gcs" {
    bucket = "interview-lab-tfstate"
    prefix = "interview-api/state"
  }
}
```
Then migrate: `terraform init -migrate-state`

## Validation
```bash
./interview validate
```
Passes when: backend type is not `local`; a recognised remote backend (gcs, s3, azurerm, etc.) is configured.

## Common Wrong Fixes (Claude traps)
- Running `terraform state push` before `terraform init -migrate-state` — wrong order; state push requires the new backend to already be initialised
- Keeping local backend and adding a `.tfvars` lock file — doesn't solve concurrent apply problem
- Adding DynamoDB locking without switching backends — partial fix; state is still local

## Follow-Up Questions
- What problems does remote state solve that local state doesn't? (Sharing, locking, auditability)
- What is the correct sequence to migrate from local to remote backend? (`init -migrate-state`, NOT state push first)
- What happens if two engineers run `terraform apply` at the same time with local backend? (State file corruption)

## Rubric Signals
- Cat 4 (Terraform): 5 if they know `terraform init -migrate-state` without prompting; 3 if they configure GCS but don't know the migration command
- Cat 11 (Judgment, senior): Claude often suggests `terraform state push` — strong candidate knows this is the wrong order
